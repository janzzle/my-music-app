# 보안 감사 스킬 (Security Audit Skill)

코드의 보안을 리뷰할 때 다음 단계를 엄격히 따르세요:

## 🛡️ 리뷰 체크리스트
1. **입력값 검증 (Input Validation)**: 사용자 입력(URL 파라미터, Form 데이터 등)이 렌더링/저장되기 전에 적절히 이스케이프(Escape)되거나 살균(Sanitize) 처리되었는가? (XSS 방어)
2. **데이터 접근 (Data Access)**: 데이터베이스 쿼리나 API 요청 시, 변수가 하드코딩되지 않고 파라미터화(Parameterized) 되어있는가? (Injection 방어)
3. **인증 및 권한 (Auth & Authorization)**: 중요 데이터나 관리자 페이지 접근 시 사용자 권한을 검증하는 로직이 누락되지 않았는가?
4. **민감 정보 노출 (Sensitive Data)**: API 키, 비밀번호, 환경 변수 등이 코드나 콘솔 로그에 평문으로 노출되어 있지 않은가?

## 📝 리포트 제공 방법
- 취약점이 발견된 **정확한 파일명과 줄 번호**를 명시하세요.
- 해당 코드가 왜 위험한지(어떤 방식의 공격이 가능한지) 설명하세요.
- 취약점을 수정할 수 있는 **안전한 코드 스니펫(대안)**을 반드시 제공하세요.
- 발견된 문제가 없다면 "보안 특이사항 없음"이라고 명확히 답변하세요.

# UI 컴포넌트 설계 스킬 (UI Component Architect Skill)

새로운 UI 컴포넌트를 설계하고 코드를 작성할 때 다음 원칙을 따르세요:

## 🎨 설계 체크리스트
1. **반응형 (Responsive)**: 모바일(Mobile-first)에서 데스크탑까지 화면 크기 변화에 대응하는 클래스(예: Tailwind의 sm:, md: 등)가 포함되어 있는가?
2. **접근성 (Accessibility - a11y)**: `aria-label`, `alt` 속성, 키보드 네비게이션(Tab 인덱스) 등 스크린 리더 사용자를 위한 처리가 되어 있는가?
3. **상태 및 속성 (State & Props)**: 컴포넌트의 Props 인터페이스가 명확히 정의되어 있으며, 불필요한 내부 상태(Local State)를 남용하지 않았는가?
4. **재사용성 (Reusability)**: 색상이나 폰트 크기 등이 하드코딩되지 않고, 프로젝트의 전역 테마(Theme) 변수를 활용하고 있는가?

## 🛠️ 컴포넌트 작성 의사결정 트리
- **이 컴포넌트가 서버 데이터를 가져와야 하는가?**
  - **Yes**: 데이터 패칭 로직(Container)과 화면 렌더링 로직(Presentational)을 분리하여 작성하세요.
  - **No**: 순수 UI 렌더링에 집중한 함수형 컴포넌트로 작성하세요.

## 📝 결과물 제공 방법
- 컴포넌트의 전체 코드를 제공하세요.
- 복잡한 로직이나 접근성 처리를 한 부분에는 주석을 달아 이유를 설명하세요.

# 리팩토링 및 클린 코드 스킬 (Refactoring Skill)

코드 리팩토링을 진행할 때 다음 단계를 따르세요:

## 🧹 리팩토링 체크리스트
1. **단일 책임 원칙 (Single Responsibility)**: 하나의 함수/컴포넌트가 너무 많은 일을 하고 있지 않은가? (그렇다면 여러 개의 작은 함수로 분리하세요)
2. **네이밍 컨벤션 (Naming)**: 변수명과 함수명이 그 목적과 역할을 직관적이고 명확하게 설명하고 있는가? (축약어 사용 자제)
3. **중복 제거 (DRY - Don't Repeat Yourself)**: 반복적으로 사용되는 로직이 있다면 공통 유틸리티 함수나 훅(Hook)으로 분리했는가?
4. **조기 반환 (Early Return)**: 중첩된 if-else 문(Depth)이 너무 깊지 않은가? (조건이 맞지 않으면 함수 초반에 미리 `return` 처리하세요)

## ⚠️ 실행 시 주의사항 (Black Box 원칙)
- 절대로 기존의 **비즈니스 로직(결과값)을 변경해서는 안 됩니다.**
- 만약 프로젝트 내에 테스트 스크립트(`npm test` 등)가 존재한다면, 코드를 수정한 후 스크립트를 실행하여 테스트가 깨지지 않았는지 스스로 검증하세요.

## 📝 피드백 제공 방법
- 수정 전/후의 변경 사항을 설명하고, 왜 이 구조가 더 나은지(Why) 짧게 코멘트하세요.
